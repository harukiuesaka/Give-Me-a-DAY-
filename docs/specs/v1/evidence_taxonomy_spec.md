# Give Me a DAY v1 — Evidence Taxonomy Specification

**Document type**: Internal system specification
**Domain**: Investment research / Strategy validation / Hypothesis-testing pipelines
**Version**: v1 draft
**Status**: Design phase — pre-implementation
**Upstream dependencies**: v1_core_loop_spec.md, v1_audit_rubric_spec.md, v1_output_package_spec.md

---

## Purpose

Evidence Taxonomy は「候補の検証に使うデータの分類辞書」である。

Audit Rubric の `evidence_gap` (EVD-01〜06) と `leakage_risk` (LKG-01〜07) は「どの種類のデータに対して」走査するかが未定義のままでは実装できない。本仕様はその不足を埋める。

Taxonomy の責務は3つ:

1. **分類**: 投資リサーチで使用されるデータを、品質特性・バイアス特性・リーケージ特性が類似するグループに分ける
2. **品質基準の提供**: 各カテゴリに対し、品質評価軸・バイアスリスク・proxy 許容条件を定義する
3. **Audit / Validation との接続**: 各カテゴリが Audit Rubric のどのパターンに、Validation Plan のどのテストに接続するかを明示する

Taxonomy は「データカタログ」ではない。個別データセットの網羅的リストではなく、**カテゴリレベルの品質判断フレームワーク**である。

---

## Taxonomy 全体構造

```
Evidence Taxonomy
├── [PRC] Price Data          ── 価格・出来高・約定データ
├── [FND] Fundamental Data    ── 財務諸表・企業業績データ
├── [ALT] Alternative Data    ── 非伝統的・非構造化データ
├── [MAC] Macro Data          ── マクロ経済指標データ
├── [SNT] Sentiment Data      ── 市場心理・センチメントデータ
├── [FLW] Flow Data           ── 資金フロー・ポジションデータ
└── [MTA] Metadata            ── 銘柄属性・分類・構造データ
```

各カテゴリの境界原則: **あるデータが2つのカテゴリに属しうる場合、そのデータのバイアス特性とリーケージ特性がより深刻なカテゴリに分類する。** 例: アナリストの業績予想は FND と SNT の境界にあるが、リーケージ特性（公表タイミングの不確実性）が SNT 寄りのため SNT に分類する。

## [PRC] Price Data — 価格・出来高・約定データ

### 1. 定義
金融商品の市場価格、出来高、約定に関する時系列データ。始値・高値・安値・終値 (OHLC)、出来高、VWAP、ビッド・アスク、約定履歴を含む。頻度は日次からティックレベルまで。

### 2. このカテゴリが重要な理由
投資戦略のバックテストの基盤。ほぼ全ての候補が直接的または間接的に Price Data に依存する。Price Data の品質問題は他の全カテゴリのデータ評価にも波及する（リターン計算、ボラティリティ推定等の基礎として使用されるため）。

### 3. 代表的なデータ例
- 日次 OHLCV（始値・高値・安値・終値・出来高）
- 分足・秒足の価格データ
- ビッド・アスクスプレッド
- 約定履歴（time & sales）
- 調整済み終値（配当・分割調整後）
- 先物・オプションの価格データ
- 指数の構成銘柄ウェイト付きリターン

### 4. 典型的ソース

| ソース種別 | 例 | 品質帯 | コスト帯 |
|-----------|---|--------|---------|
| プレミアムベンダー | Bloomberg, Refinitiv, FactSet | 高 | 高 |
| 取引所直接配信 | JPX, NYSE, NASDAQ | 高 | 中〜高 |
| 中間ベンダー | Quandl/Nasdaq Data Link, Norgate | 中〜高 | 中 |
| 無料ソース | Yahoo Finance, Google Finance | 低〜中 | 無料 |
| ブローカーAPI | Interactive Brokers, Alpaca | 中 | 低〜中 |

### 5. 品質評価軸

| 品質軸 | 評価基準 | 投資リサーチでの重要度 |
|--------|---------|-------------------|
| 時間精度 | タイムスタンプの粒度と正確性。日次なら十分か、分足が必要か | 戦略の時間軸に依存。高頻度戦略では critical |
| 調整の正確性 | 株式分割・配当・併合等のコーポレートアクション調整が正しいか | critical。未調整データでのバックテストは無意味 |
| 欠損率 | 欠損値の割合。取引停止日の扱い | medium。欠損処理方法により結果が変動 |
| 生存者バイアスの有無 | 上場廃止銘柄のデータが含まれるか | critical。特にユニバース構築に使用する場合 |
| 配信遅延 | リアルタイム性。リアルタイムか遅延か EOD か | 戦略の時間軸に依存 |
| ソースの一貫性 | 同一ソースの長期データで方法論が変更されていないか | medium。方法論変更は構造変化として検出が必要 |

### 6. バイアス / 汚染 / 時間ずれリスク

| リスク ID | リスク名 | 説明 | 深刻度 |
|----------|---------|------|--------|
| PRC-B01 | Survivorship bias | 上場廃止銘柄が除外されたユニバースで価格データを取得 | critical |
| PRC-B02 | 調整バイアス | コーポレートアクション調整の誤りまたは不完全性 | high |
| PRC-B03 | 閉場価格の不正確性 | 薄商い銘柄の終値が実際の取引可能価格と乖離 | medium |
| PRC-B04 | VWAP 計算の不整合 | ソースによる VWAP 計算方法の違い | medium |
| PRC-B05 | 時間帯の不整合 | 複数市場のデータで取引時間帯のマッピングが不正確 | high（クロスマーケット戦略） |
| PRC-B06 | Backfill | 過去データの事後修正がリアルタイムのデータと異なる | medium |

### 7. Leakage risk の典型パターン

| Leakage Pattern | Audit Rubric 接続 | 検出方法 |
|----------------|-------------------|---------|
| 当日終値を当日のシグナル生成に使用 | LKG-01 (look-ahead) | シグナル生成タイムスタンプ vs データ利用可能タイムスタンプの比較 |
| 調整済み終値に将来のコーポレートアクション情報が含まれる | LKG-01 (look-ahead) | 調整済みデータの再計算タイミングを確認 |
| 上場廃止銘柄を含まないユニバースでの検証 | LKG-02 (survivorship) | ユニバース構築方法の確認。point-in-time ユニバースかどうか |
| 最適な lookback period を全期間のデータで選定 | LKG-03 (data snooping) | パラメータ選択に使用したデータと検証データの分離確認 |

### 8. Proxy として許容できる条件
- 日次 OHLCV が必要だが終値のみ利用可能 → **許容**。ただし、高値・安値に基づくボラティリティ推定（Parkinson等）が必要な候補では不可
- 個別銘柄データが必要だが指数レベルのデータで代替 → **条件付き許容**。候補が指数レベルの検証で十分な場合のみ。銘柄選択戦略では不可
- ティックデータが必要だが分足で代替 → **許容**。ただし market microstructure に依存する候補では不可。quality_loss_estimate: medium

### 9. Proxy として許容できない条件
- 調整済み終値が必要だが未調整終値で代替 → **不可**。コーポレートアクション未調整のリターン計算は根本的に誤り
- point-in-time ユニバースが必要だが現在の指数構成銘柄で代替 → **不可**。survivorship bias が直接混入
- 出来高データが必要だが「出来高なし」で代替（出来高をゼロまたは無限と仮定）→ **不可**。流動性制約の検証が不能

### 10. v1 で最低限検査すべき項目
- [ ] コーポレートアクション調整の有無と方法
- [ ] survivorship bias の有無（ユニバースに上場廃止銘柄を含むか）
- [ ] データの時間粒度が候補の要求を満たすか
- [ ] 欠損値の処理方法が明示されているか
- [ ] 複数市場のデータを使用する場合、時間帯の整合性

### 11. Audit Rubric との接続

| Audit Pattern | Price Data での検出対象 |
|--------------|----------------------|
| EVD-01 (必須証拠未入手) | 候補が必要とする価格データの availability 確認 |
| EVD-02 (時間的不足) | 価格データの temporal_coverage が Validation Plan の要求期間をカバーするか |
| EVD-04 (バイアス未検証) | PRC-B01〜B06 が known_biases に記録されているか。影響度評価の有無 |
| LKG-01 (look-ahead) | 当日終値のシグナル利用、将来調整値の使用 |
| LKG-02 (survivorship) | ユニバース構築における上場廃止銘柄の扱い |
| LKG-07 (point-in-time) | 調整済みデータの再計算タイミング |
| RLM-01 (取引コスト) | ビッド・アスクスプレッドデータの有無（コスト推定の基礎） |
| RLM-02 (流動性) | 出来高データの有無（流動性制約の検証基礎） |

### 12. Validation Plan との接続

| Validation Test | Price Data の役割 |
|----------------|------------------|
| offline_backtest | OHLCV が必須入力。調整済み終値でリターン計算 |
| walk_forward | 時系列の分割に使用。temporal_coverage が十分か |
| out_of_sample | サンプル外期間の価格データが確保されているか |
| stress_test | 危機期間（2008年、2020年3月等）の価格データを含むか |
| regime_split | レジーム判定に VIX 等のボラティリティ指標データが必要 |
| sensitivity | 取引コスト感度分析にスプレッドデータが必要 |

## [FND] Fundamental Data — 財務諸表・企業業績データ

### 1. 定義
企業の財務諸表（損益計算書・貸借対照表・キャッシュフロー計算書）および派生指標（PER, PBR, ROE 等）。決算短信、有価証券報告書、アニュアルレポートから取得。頻度は四半期または年次。

### 2. このカテゴリが重要な理由
バリュー投資、ファンダメンタル分析、クオリティファクター戦略の基盤。ファンダメンタルデータは「公表タイミング」と「利用可能タイミング」のずれが最も深刻なカテゴリであり、leakage の温床。

### 3. 代表的なデータ例
- 売上高、営業利益、純利益、EPS
- 総資産、純資産、自己資本比率
- 営業CF、投資CF、財務CF
- PER、PBR、配当利回り、ROE、ROA
- 設備投資額、研究開発費
- セグメント別売上・利益
- 発行済株式数、自己株式数

### 4. 典型的ソース

| ソース種別 | 例 | 品質帯 | コスト帯 |
|-----------|---|--------|---------|
| プレミアムベンダー | Bloomberg, FactSet, S&P Capital IQ | 高 | 高 |
| データベンダー | Compustat, Worldscope | 高 | 高 |
| 中間ベンダー | Quandl/Nasdaq Data Link (Sharadar) | 中 | 中 |
| 公開ソース | EDINET (日本), EDGAR (米国) | 中（生データ） | 無料 |
| 無料集約 | 各種スクリーナー | 低〜中 | 無料 |

### 5. 品質評価軸

| 品質軸 | 評価基準 | 投資リサーチでの重要度 |
|--------|---------|-------------------|
| Point-in-time 対応 | データが「いつ公表されたか」のタイムスタンプを保持するか | critical。これがないと LKG-06/07 が不可避 |
| 修正値の扱い | 決算修正・再表示の履歴が保持されるか | critical。修正後の値のみ保持するソースは look-ahead を生む |
| 会計基準の一貫性 | IFRS / US GAAP / JP GAAP の混在がないか | high。クロスカントリー比較で致命的 |
| 欠損処理 | 未報告企業の扱い。欠損=0か欠損=nullか | medium。処理方法で指標値が大幅に変動 |
| セグメント粒度 | 連結 vs 単体、セグメント別の有無 | 候補による。コングロマリット分析には必須 |
| 通貨統一 | 多国籍企業の通貨換算方法 | high。為替効果の混入 |

### 6. バイアス / 汚染 / 時間ずれリスク

| リスク ID | リスク名 | 説明 | 深刻度 |
|----------|---------|------|--------|
| FND-B01 | 報告ラグ無視 | 3月期決算が6月に公表される事実を無視し、4月1日から利用可能と仮定 | critical |
| FND-B02 | 修正値の遡及適用 | 修正された財務数値を、修正前の時点で利用可能だったかのように扱う | critical |
| FND-B03 | Survivorship bias | 倒産・上場廃止企業のファンダメンタルが除外 | high |
| FND-B04 | Backfill bias | データベンダーが新規カバレッジ開始時に過去データを遡及追加 | high |
| FND-B05 | 会計方針変更 | 同一企業の会計基準変更による時系列の断絶 | medium |
| FND-B06 | 季節性の不整合 | 決算月が企業間で異なることによる比較の歪み | medium |

### 7. Leakage risk の典型パターン

| Leakage Pattern | Audit Rubric 接続 | 検出方法 |
|----------------|-------------------|---------|
| 未公表の決算データをシグナル生成に使用 | LKG-01 (look-ahead) | シグナル生成日 vs 決算公表日の比較。公表日が未記録なら flag |
| 修正後の財務数値を修正前の時点で使用 | LKG-07 (point-in-time) | point-in-time 対応のデータソースを使用しているか確認 |
| 決算修正の予知 | LKG-01 (look-ahead) | 修正後の値のみ保持するデータソースでのバックテスト |
| 報告ラグを無視したファクター計算 | LKG-06 (報告ラグ) | ファクター更新タイミングと決算公表スケジュールの照合 |

### 8. Proxy として許容できる条件
- 四半期データが必要だが年次データで代替 → **条件付き許容**。低頻度リバランス戦略（年次以下）では許容。quality_loss_estimate: medium
- 個別企業データが必要だがセクター集計値で代替 → **条件付き許容**。セクターローテーション戦略では許容。銘柄選択戦略では不可
- Point-in-time データが必要だが公表日カラムのみで代替 → **条件付き許容**。公表日に十分なバッファ（例: 決算月末 + 90日）を加える場合。ただし LKG-06 リスクを Audit で明記

### 9. Proxy として許容できない条件
- Point-in-time データが必要だが修正後の最新値のみで代替 → **不可**。look-ahead bias が直接混入。バイアスの方向と大きさが予測不能
- 実績値が必要だがアナリスト予想値（コンセンサス）で代替 → **不可**。予想と実績は根本的に異なるデータ。予想はSNTカテゴリに属する
- 日本企業の JP GAAP データが必要だが IFRS 換算値で代替 → **不可**。会計基準間の差異が大きい項目（のれん、リース等）で系統的な歪みが発生

### 10. v1 で最低限検査すべき項目
- [ ] point-in-time 対応の有無。未対応の場合、報告ラグのバッファ設定方法
- [ ] 決算公表日のデータが利用可能か
- [ ] 修正値の扱い（最新値のみか、修正履歴を保持するか）
- [ ] survivorship bias の有無（倒産・上場廃止企業のカバレッジ）
- [ ] 会計基準の統一性（複数国を扱う場合）

### 11. Audit Rubric との接続

| Audit Pattern | Fundamental Data での検出対象 |
|--------------|---------------------------|
| EVD-01 | 候補のファクター計算に必要な財務項目の availability |
| EVD-04 | FND-B01〜B06 の known_biases 記録の有無 |
| LKG-01 | 未公表決算データのシグナル利用 |
| LKG-06 | 報告ラグの無視 |
| LKG-07 | point-in-time 違反 |
| ASM-04 | データ品質の暗黙仮定（修正値の遡及適用を意識していない等） |

### 12. Validation Plan との接続

| Validation Test | Fundamental Data の役割 |
|----------------|----------------------|
| offline_backtest | ファクター値の計算入力。報告ラグを反映した利用可能日の設定が必須 |
| walk_forward | ファクター更新タイミングが walk-forward の分割と整合するか |
| out_of_sample | サンプル外期間に十分なファンダメンタルデータが存在するか |
| regime_split | 業績サイクル（好況/不況）とレジーム分割の整合性 |
| sensitivity | ファクター定義の微小変更（PER の計算方法等）に対する感度 |

## [ALT] Alternative Data — 非伝統的・非構造化データ

### 1. 定義
伝統的な価格データ・財務データ以外の、投資判断に利用されるデータ。衛星画像、Webスクレイピング、クレジットカード決済、アプリ利用、求人情報、特許データ等。構造化の程度が低く、品質のばらつきが大きい。

### 2. このカテゴリが重要な理由
アルファの源泉として注目度が高いが、品質管理が最も困難。歴史が浅くバックテスト期間が短い、ベンダー間の標準化がない、backfill bias が深刻、データの継続性が保証されない、という特性がある。「新しいデータ = 優位性」という誤解が false confidence を生みやすい。

### 3. 代表的なデータ例
- 衛星画像（駐車場の車数、農作物の生育状況）
- クレジットカード・POS トランザクション
- Web トラフィック・アプリダウンロード数
- ソーシャルメディア投稿量・内容
- 求人情報の増減
- 特許出願データ
- サプライチェーンデータ（船舶追跡、コンテナ量）
- 政府公開データの早期推定（雇用、消費等）

### 4. 典型的ソース

| ソース種別 | 例 | 品質帯 | コスト帯 |
|-----------|---|--------|---------|
| 専門ベンダー | Orbital Insight, Quandl, Eagle Alpha | 中〜高 | 高 |
| データマーケットプレイス | Snowflake Marketplace, AWS Data Exchange | ばらつき大 | 中〜高 |
| 自社スクレイピング | 独自構築 | 低〜中 | 低（構築コスト高） |
| 公開データ | 政府統計の早期推定版 | 中 | 無料〜低 |

### 5. 品質評価軸

| 品質軸 | 評価基準 | 投資リサーチでの重要度 |
|--------|---------|-------------------|
| 歴史の長さ | バックテストに十分な期間のデータがあるか | critical。短い歴史は overfitting の温床 |
| 継続性保証 | ベンダーのデータ提供が将来も継続する保証があるか | high。データ打ち切りで戦略が実行不能になるリスク |
| カバレッジの安定性 | 時間経過でカバレッジ（対象企業数等）が変化していないか | critical。カバレッジ拡大は backfill bias を生む |
| 変換・加工の透明性 | 生データから提供データへの変換過程が文書化されているか | high。ブラックボックス変換は検証不能 |
| サンプリングバイアス | データの収集対象が母集団を代表しているか | high。例: クレジットカードデータは富裕層に偏る |
| 再現性 | 同一クエリで同一結果が得られるか。過去データの改変がないか | critical。再現不能なデータは検証不能 |

### 6. バイアス / 汚染 / 時間ずれリスク

| リスク ID | リスク名 | 説明 | 深刻度 |
|----------|---------|------|--------|
| ALT-B01 | Backfill bias（深刻） | ベンダーがカバレッジを拡大した際、過去データを遡及追加。過去のカバレッジが実際より広く見える | critical |
| ALT-B02 | サンプリングバイアス | データ収集対象が特定の人口統計・地域・企業規模に偏る | high |
| ALT-B03 | 方法論変更 | ベンダーのデータ収集・処理方法が途中で変更され、時系列の一貫性が崩壊 | high |
| ALT-B04 | 季節性の人為的発生 | データ収集の技術的制約が季節パターンを生む（例: 衛星画像の雲量） | medium |
| ALT-B05 | 遅延の不安定性 | データの配信遅延が時期によって変動。バックテストでは最短遅延を仮定しがち | high |
| ALT-B06 | 情報の先行織り込み | オルタナティブデータが広く利用されるようになった時点でアルファが消失 | high |

### 7. Leakage risk の典型パターン

| Leakage Pattern | Audit Rubric 接続 | 検出方法 |
|----------------|-------------------|---------|
| ベンダーの backfill データを過去時点で利用可能と仮定 | LKG-05 (backfill) | ベンダーのカバレッジ開始日と、バックテスト開始日を比較。バックテスト開始日 < カバレッジ開始日なら flag |
| データ配信遅延を無視し、速報値をリアルタイムで利用可能と仮定 | LKG-01 (look-ahead) | データの実配信タイミングとシグナル生成タイミングの照合 |
| ベンダーが事後修正したデータを修正前の時点で使用 | LKG-07 (point-in-time) | データバージョン管理の有無を確認 |
| データ方法論変更の事後知識で「変更後のみ」を使用 | LKG-03 (data snooping) | 方法論変更日とバックテスト期間の照合 |

### 8. Proxy として許容できる条件
- 特定ベンダーのオルタナティブデータが必要だが、類似の公開データで代替 → **条件付き許容**。カバレッジと品質の差を quality_loss_estimate として定量化する場合のみ
- 日次データが必要だが週次集計で代替 → **条件付き許容**。低頻度戦略（週次以下のリバランス）の場合

### 9. Proxy として許容できない条件
- backfill を含むデータで「backfill がない」と仮定して代替 → **不可**。backfill の存在自体がデータの品質を根本的に変える
- カバレッジ開始前の期間を「データなし = 中立シグナル」として代替 → **不可**。データ不在とシグナル中立は異なる概念
- あるベンダーのオルタナティブデータを、方法論が異なる別ベンダーのデータで代替 → **不可**。方法論の差異が系統的バイアスを生む

### 10. v1 で最低限検査すべき項目
- [ ] データの歴史の長さ（バックテスト期間に対して十分か）
- [ ] backfill の有無。有の場合、backfill 前の実カバレッジ範囲
- [ ] ベンダーのデータ方法論が文書化されているか
- [ ] データ配信の遅延（公称値と実測値）
- [ ] カバレッジの時系列変化（拡大・縮小の有無）

### 11. Audit Rubric との接続

| Audit Pattern | Alternative Data での検出対象 |
|--------------|---------------------------|
| EVD-01 | 必要なオルタナティブデータの入手可能性 |
| EVD-02 | 歴史の短さ。バックテスト期間の十分性 |
| EVD-04 | ALT-B01〜B06 の known_biases 記録の有無 |
| LKG-05 | backfill bias |
| LKG-01 | 配信遅延の無視 |
| OVF-01 | 短い歴史でのパラメータ過多（サンプルサイズ不足） |
| ASM-06 | 単一ベンダー依存 |

### 12. Validation Plan との接続

| Validation Test | Alternative Data の役割 |
|----------------|----------------------|
| offline_backtest | backfill を除外した期間でのみ実施すべき。backfill 期間を含むバックテストは無効 |
| out_of_sample | オルタナティブデータの歴史が短い場合、サンプル外データの確保が困難。この制約を plan_completeness に反映 |
| sensitivity | データベンダー変更・方法論変更に対する感度分析 |
| stress_test | データが利用不能になったシナリオ（ベンダー撤退等）の影響評価 |

## [MAC] Macro Data — マクロ経済指標データ

### 1. 定義
国・地域レベルの経済指標。GDP、雇用統計、インフレ率、金利、為替レート、中央銀行政策等。公的機関が定期的に公表し、改定が複数回行われるのが特徴。

### 2. このカテゴリが重要な理由
マクロ戦略・グローバルマクロ・リスクマネジメントの基盤。レジーム判定にも使用される。最大の固有リスクは「改定」——初回発表値と最終確定値が大幅に異なる場合があり、確定値を初回発表時点で使用すると深刻な look-ahead bias が生じる。

### 3. 代表的なデータ例
- GDP（速報・改定・確定）
- 非農業部門雇用者数（NFP）
- CPI / PCE デフレーター
- 政策金利、イールドカーブ
- 為替レート
- 購買担当者指数（PMI）
- 貿易収支、経常収支
- マネーサプライ（M1, M2）

### 4. 典型的ソース

| ソース種別 | 例 | 品質帯 | コスト帯 |
|-----------|---|--------|---------|
| 公的機関直接 | FRED, 日銀, ECB, BLS, 内閣府 | 高 | 無料 |
| プレミアムベンダー | Bloomberg, Refinitiv | 高（point-in-time対応） | 高 |
| データ集約サービス | OECD.Stat, World Bank | 中〜高 | 無料 |
| リアルタイム改定DB | ALFRED (FRED のpoint-in-time版) | 高 | 無料 |

### 5. 品質評価軸

| 品質軸 | 評価基準 | 投資リサーチでの重要度 |
|--------|---------|-------------------|
| ビンテージ対応 | 各時点で利用可能だったデータバージョン（ビンテージ）が保持されているか | critical。改定のある指標では必須 |
| 改定履歴 | 初回発表→速報→改定→確定の各値と日付が保持されているか | critical |
| 季節調整方法 | 季節調整の方法と改定の有無。季節調整は遡及修正される | high |
| 公表スケジュール | 各指標の公表日カレンダーが利用可能か | high。イベントドリブン戦略では必須 |
| 基準年変更 | GDP デフレーター等の基準年変更による時系列の断絶 | medium |

### 6. バイアス / 汚染 / 時間ずれリスク

| リスク ID | リスク名 | 説明 | 深刻度 |
|----------|---------|------|--------|
| MAC-B01 | ビンテージ無視 | 確定値を初回発表時点で利用可能と仮定。GDP初回と確定で1%以上のずれは珍しくない | critical |
| MAC-B02 | 季節調整の遡及修正 | 季節調整係数が毎年改定され、過去のデータが変わる | high |
| MAC-B03 | 公表タイミングの仮定 | 公表日ではなく参照期間の末日にデータが利用可能と仮定 | critical |
| MAC-B04 | 基準年変更の無視 | 基準年変更前後のデータを接続せずに使用 | medium |
| MAC-B05 | 国際比較の非同期性 | 各国の公表タイミングが異なることによる情報の非対称性を無視 | medium |

### 7. Leakage risk の典型パターン

| Leakage Pattern | Audit Rubric 接続 | 検出方法 |
|----------------|-------------------|---------|
| GDP確定値を速報発表日に使用 | LKG-07 (point-in-time) | 使用しているデータが ALFRED 等のビンテージ対応ソースか確認 |
| 雇用統計の改定後値を改定前の時点で使用 | LKG-07 (point-in-time) | 改定履歴の有無。改定なしのソースでマクロ戦略はリーケージ不可避 |
| 季節調整後データの遡及修正を無視 | LKG-01 (look-ahead) | 季節調整の改定がデータに反映されているか |
| 公表カレンダーの事後知識を使用 | LKG-01 (look-ahead) | サプライズを事前に知っているかのような売買タイミング |

### 8. Proxy として許容できる条件
- ビンテージ対応データが必要だが公表日 + バッファ（例: +30日）で代替 → **条件付き許容**。バッファが改定サイクルをカバーする場合。quality_loss_estimate: medium。改定幅が大きい指標（GDP等）では非推奨
- 月次データが必要だが四半期データで代替 → **条件付き許容**。低頻度戦略のみ
- 特定国のデータが必要だが類似経済圏のデータで代替 → **条件付き許容**。マクロ環境の類似性を定量的に示せる場合のみ

### 9. Proxy として許容できない条件
- ビンテージ対応データが必要だが確定値のみで代替 → **不可**。GDP等の改定幅が大きい指標では、確定値の使用は look-ahead bias そのもの
- 速報値が必要だが「速報値と確定値はほぼ同じだろう」と仮定して確定値で代替 → **不可**。平均的には差が小さくても、大きな乖離が生じる局面こそが戦略の転換点になる
- 国別データが必要だがグローバル集計値で代替 → **不可**。国固有の政策・サイクルの差異が消失

### 10. v1 で最低限検査すべき項目
- [ ] 改定のある指標でビンテージ対応のソースを使用しているか
- [ ] 公表日データが利用可能か（データがいつから使えたかの記録）
- [ ] 季節調整方法の記録があるか
- [ ] 複数国のデータを使用する場合、公表タイミングの非同期性を考慮しているか
- [ ] 確定値と速報値の使い分けが明示されているか

### 11. Audit Rubric との接続

| Audit Pattern | Macro Data での検出対象 |
|--------------|----------------------|
| EVD-01 | マクロ指標の入手可能性（特にビンテージ対応版） |
| LKG-01 | 季節調整の遡及修正、公表カレンダーの事後知識 |
| LKG-07 | ビンテージ無視。確定値の遡及利用 |
| RGM-04 | マクロ環境依存が明示されているか。使用するマクロ指標とレジーム判定の整合性 |
| ASM-02 | マクロ環境の定常性仮定。過去の金利環境が将来も継続する仮定等 |

### 12. Validation Plan との接続

| Validation Test | Macro Data の役割 |
|----------------|-----------------|
| offline_backtest | マクロファクターの入力。ビンテージ対応データの使用が必須 |
| regime_split | レジーム判定の入力。金利環境・インフレ環境・成長環境等 |
| stress_test | マクロショックシナリオの構築入力（金利急騰、景気後退等） |
| sensitivity | マクロ指標の定義変更（季節調整方法、基準年等）に対する感度 |

## [SNT] Sentiment Data — 市場心理・センチメントデータ

### 1. 定義
市場参加者の心理・期待・見通しを反映するデータ。アナリスト予想、ニュースセンチメント、投資家サーベイ、オプション市場から推定されるインプライドボラティリティ等。定量化の方法がソースにより大きく異なる。

### 2. このカテゴリが重要な理由
行動ファイナンス系戦略、イベントドリブン戦略、ニュースベース戦略の基盤。最大の固有リスクは「定量化の恣意性」——同一のニュースでもセンチメントスコアがベンダーにより大幅に異なる。また、公表タイミングの不確実性が高い（アナリストレポートの発行時刻 ≠ 市場への伝播時刻）。

### 3. 代表的なデータ例
- アナリスト業績予想（コンセンサス、個別）
- アナリストレーティング（アップグレード/ダウングレード）
- ニュースセンチメントスコア
- ソーシャルメディアセンチメント
- 投資家サーベイ（AAII, Investors Intelligence 等）
- VIX / インプライドボラティリティ
- put/call レシオ
- インサイダー取引情報

### 4. 典型的ソース

| ソース種別 | 例 | 品質帯 | コスト帯 |
|-----------|---|--------|---------|
| コンセンサスDB | I/B/E/S, Bloomberg, FactSet | 高 | 高 |
| NLPベンダー | RavenPack, Alexandria, Refinitiv News Analytics | 中〜高 | 高 |
| ソーシャル分析 | StockTwits, Twitter API | 低〜中 | 低〜中 |
| 公開サーベイ | AAII, Investors Intelligence | 中 | 低 |
| 市場データ派生 | CBOE (VIX), options chains | 高 | 低〜中 |

### 5. 品質評価軸

| 品質軸 | 評価基準 | 投資リサーチでの重要度 |
|--------|---------|-------------------|
| スコアリング一貫性 | 同一ベンダーのセンチメントスコアが時間経過で一貫しているか | critical。アルゴリズム変更によるスコア断絶 |
| 公表タイミングの精度 | アナリストレポート・ニュースの正確な発信時刻が利用可能か | critical。分単位の精度が必要な場合あり |
| カバレッジの偏り | 大型株に偏り、小型株のカバレッジが薄くないか | high |
| ノイズレベル | シグナル対ノイズ比。特にソーシャルメディアデータ | high |
| 再現性 | NLP ベンダーのスコアが後日再計算可能か | medium |

### 6. バイアス / 汚染 / 時間ずれリスク

| リスク ID | リスク名 | 説明 | 深刻度 |
|----------|---------|------|--------|
| SNT-B01 | アナリスト予想の公表タイミング不確実性 | 予想の更新がいつ市場に反映されたかが不明確 | critical |
| SNT-B02 | NLPスコアの遡及改変 | ベンダーがアルゴリズム更新時にスコアを遡及再計算 | critical |
| SNT-B03 | 大型株偏重 | アナリストカバレッジが大型株に集中。小型株のセンチメント情報が不足 | high |
| SNT-B04 | ソーシャルメディアのボット汚染 | 自動投稿・操作的投稿がセンチメントを歪める | high |
| SNT-B05 | 生存者バイアス（アナリスト） | カバレッジ停止企業の過去予想が除外される | medium |
| SNT-B06 | ハーディング効果 | アナリスト間の群集行動により、独立した情報源に見えて実は相関が高い | medium |

### 7. Leakage risk の典型パターン

| Leakage Pattern | Audit Rubric 接続 | 検出方法 |
|----------------|-------------------|---------|
| アナリスト予想の公表前にシグナル生成 | LKG-01 (look-ahead) | 予想更新タイムスタンプとシグナル生成日の照合 |
| NLPスコアの遡及再計算値をバックテストに使用 | LKG-05 (backfill) + LKG-07 (point-in-time) | ベンダーのスコア改定ポリシーの確認 |
| コンセンサス予想のデータベースで最終修正値を過去時点で使用 | LKG-07 (point-in-time) | point-in-time コンセンサスDB（I/B/E/S Detail History等）の使用確認 |
| 決算後のアナリストリビジョンを決算前のシグナルに使用 | LKG-01 (look-ahead) | リビジョン日と決算発表日の前後関係 |

### 8. Proxy として許容できる条件
- コンセンサス予想が必要だが個別アナリスト予想の中央値で代替 → **許容**。ただしアナリスト数が十分（≥5）な場合のみ
- ニュースセンチメントが必要だがソーシャルメディアセンチメントで代替 → **条件付き許容**。大型株で両者の相関が高い場合のみ。小型株では不可。quality_loss_estimate: high
- VIXが必要だが実現ボラティリティで代替 → **条件付き許容**。リスクオン/オフの判定用途では許容。インプライドボラティリティの水準そのものが重要な候補では不可

### 9. Proxy として許容できない条件
- point-in-time コンセンサスが必要だが最新コンセンサスで代替 → **不可**。アナリストリビジョンのタイミング情報が消失
- NLP スコアが必要だが「ポジティブ/ネガティブ」のバイナリ判定で代替 → **不可**。スコアの連続性が失われ、戦略のシグナル品質が根本的に変わる
- 機関投資家サーベイが必要だが個人投資家サーベイで代替 → **不可**。投資家層が異なり、情報内容が根本的に異なる

### 10. v1 で最低限検査すべき項目
- [ ] センチメントスコアのソースと計算方法の文書化
- [ ] アナリスト予想使用時の point-in-time 対応
- [ ] NLP ベンダーのスコア改定ポリシー（遡及再計算の有無）
- [ ] カバレッジの偏り（大型/小型）
- [ ] タイムスタンプの精度（日次 vs 分足 vs ティック）

### 11. Audit Rubric との接続

| Audit Pattern | Sentiment Data での検出対象 |
|--------------|--------------------------|
| EVD-01 | 必要なセンチメントデータの入手可能性 |
| EVD-04 | SNT-B01〜B06 の known_biases 記録の有無 |
| LKG-01 | アナリスト予想・ニュースの公表前利用 |
| LKG-05 | NLP スコアの遡及再計算 |
| LKG-07 | point-in-time コンセンサスの不使用 |
| ASM-05 | センチメントと株価の因果関係誤認（同時性・逆因果） |
| ASM-06 | 単一NLPベンダー依存 |

### 12. Validation Plan との接続

| Validation Test | Sentiment Data の役割 |
|----------------|---------------------|
| offline_backtest | センチメントシグナルの入力。point-in-time 対応必須 |
| out_of_sample | NLP スコアのアルゴリズム変更前後でのサンプル外検証 |
| sensitivity | ベンダー変更・スコアリング方法変更に対する感度 |
| regime_split | センチメント指標自体がレジーム判定に使用される場合の循環性チェック |

## [FLW] Flow Data — 資金フロー・ポジションデータ

### 1. 定義
市場参加者の資金移動・ポジション構築に関するデータ。投信の資金流出入、ETF の設定/解約、先物のポジション（COT）、信用取引残高、大量保有報告等。「何が取引されているか」の裏側にある「誰がどう動いているか」の情報。

### 2. このカテゴリが重要な理由
需給分析、ポジショニング分析、コントラリアン戦略の基盤。フローデータは遅延が構造的に大きい（報告義務の期限まで公表されない）ため、リーケージリスクが独特。また、報告の粒度が粗く、解釈の幅が大きい。

### 3. 代表的なデータ例
- CFTC Commitments of Traders (COT)
- 投信資金フロー（ICI, Morningstar, Lipper）
- ETF 設定/解約データ
- 信用取引残高（買い残・売り残）
- 大量保有報告（5%ルール）
- 機関投資家の 13F ファイリング（米国）
- 外国人投資家の売買動向

### 4. 典型的ソース

| ソース種別 | 例 | 品質帯 | コスト帯 |
|-----------|---|--------|---------|
| 規制当局 | CFTC, SEC (13F), 金融庁 | 高（公式データ） | 無料 |
| データベンダー | Bloomberg, Refinitiv | 高 | 高 |
| 業界団体 | ICI, 投資信託協会 | 中〜高 | 無料〜低 |
| 取引所 | JPX（信用取引残高） | 高 | 無料〜低 |

### 5. 品質評価軸

| 品質軸 | 評価基準 | 投資リサーチでの重要度 |
|--------|---------|-------------------|
| 報告遅延 | データが市場に公開されるまでの遅延日数 | critical。COTは最大3営業日遅延、13Fは最大45日遅延 |
| 分類精度 | 投資家カテゴリの分類が正確か（ヘッジファンド vs 商業筋等） | high。CFTC の分類変更が時系列を断絶 |
| 集計粒度 | 個別主体 vs 集計レベル | high。13F は個別、COT は集計 |
| 報告閾値 | 報告義務が発生する閾値以下の動きが見えない | medium |
| 期間の一貫性 | 報告制度の変更による時系列断絶 | medium |

### 6. バイアス / 汚染 / 時間ずれリスク

| リスク ID | リスク名 | 説明 | 深刻度 |
|----------|---------|------|--------|
| FLW-B01 | 報告遅延の構造的過小評価 | COT の金曜日データが火曜日時点のポジション。この遅延を無視するのは頻出 | critical |
| FLW-B02 | 分類変更による断絶 | CFTC が2009年に報告カテゴリを変更。変更前後のデータを連続的に扱うのは誤り | high |
| FLW-B03 | 13F の遅延と不完全性 | 45日の報告遅延。かつ全ポジションが報告対象ではない（ショートポジション除外等） | high |
| FLW-B04 | 報告閾値以下の不可視 | 小口投資家の動向が見えない | medium |
| FLW-B05 | ネッティングの歪み | グロスポジションではなくネットポジションが報告され、ヘッジ活動が見えない | medium |

### 7. Leakage risk の典型パターン

| Leakage Pattern | Audit Rubric 接続 | 検出方法 |
|----------------|-------------------|---------|
| COT データを報告基準日（火曜）にリアルタイムで利用可能と仮定 | LKG-06 (報告ラグ) | データの報告基準日と公表日の差を確認 |
| 13F データを報告期間末日に利用可能と仮定 | LKG-06 (報告ラグ) | 13F の公表日（期末+45日）を考慮しているか |
| 分類変更後のカテゴリで変更前のデータを遡及分類 | LKG-07 (point-in-time) | CFTC 分類変更日（2009年等）を跨ぐデータの扱い |

### 8. Proxy として許容できる条件
- COT データが必要だが ETF フローで代替 → **条件付き許容**。大型アセットクラス（金、原油等）で両者の相関が高い場合。quality_loss_estimate: medium
- 個別 13F データが必要だが投信フロー集計で代替 → **条件付き許容**。セクターレベルの分析であれば許容

### 9. Proxy として許容できない条件
- COT のポジショニングデータが必要だが価格データのモメンタムで代替 → **不可**。根本的に異なる情報（ポジション vs 価格変動）
- 個別機関の 13F が必要だが集計データで代替 → **不可**。個別主体の行動分析が目的の場合
- 報告遅延のあるデータをリアルタイム利用可能と仮定して代替 → **不可**。遅延はデータの本質的特性であり、仮定で除去できない

### 10. v1 で最低限検査すべき項目
- [ ] 各フローデータの報告遅延（基準日と公表日の差）
- [ ] 報告制度の変更履歴（分類変更、閾値変更等）
- [ ] シグナル生成に使用するタイミングが公表日以降であることの確認
- [ ] 報告閾値により見えない部分の影響評価

### 11. Audit Rubric との接続

| Audit Pattern | Flow Data での検出対象 |
|--------------|---------------------|
| EVD-01 | 必要なフローデータの入手可能性 |
| LKG-06 | 報告遅延の無視 |
| LKG-07 | 分類変更のpoint-in-time 違反 |
| ASM-05 | フローと価格変動の因果関係誤認 |
| RGM-04 | フローのレジーム依存性（危機時にフローパターンが激変） |

### 12. Validation Plan との接続

| Validation Test | Flow Data の役割 |
|----------------|-----------------|
| offline_backtest | フローシグナルの入力。公表日ベースの遅延を反映必須 |
| regime_split | 危機期間でのフローパターン変化の分析 |
| stress_test | 流動性危機時のフローデータの利用可能性と信頼性 |
| sensitivity | 報告遅延の仮定変更（+1日、+3日等）に対する感度 |

## [MTA] Metadata — 銘柄属性・分類・構造データ

### 1. 定義
金融商品の属性情報。セクター分類、時価総額区分、指数構成銘柄、上場/廃止日、コーポレートアクション履歴、ティッカー変更履歴等。時系列データではなく、時系列データの「座標軸」を定義するデータ。

### 2. このカテゴリが重要な理由
ユニバース構築の基盤。Metadata の品質問題は他の全カテゴリに伝播する。セクター分類が誤ればファクター分析が歪み、指数構成銘柄の変更が反映されなければ survivorship bias が生じる。見えにくいが影響は構造的。

### 3. 代表的なデータ例
- セクター・業種分類（GICS, TOPIX-17, 東証33業種）
- 時価総額区分（大型/中型/小型）
- 指数構成銘柄リスト（構成変更履歴付き）
- 上場日・廃止日
- コーポレートアクション履歴（分割、併合、配当、M&A）
- ティッカー / ISIN / SEDOL のマッピング
- 浮動株比率
- 取引所・市場区分

### 4. 典型的ソース

| ソース種別 | 例 | 品質帯 | コスト帯 |
|-----------|---|--------|---------|
| 指数提供者 | MSCI, FTSE Russell, S&P, JPX | 高 | 中〜高 |
| 分類提供者 | MSCI (GICS), ICB | 高 | 高 |
| データベンダー | Bloomberg, Refinitiv, FactSet | 高 | 高 |
| 取引所 | JPX, NYSE | 中〜高 | 低〜中 |
| 公開データ | EDINET, EDGAR | 中 | 無料 |

### 5. 品質評価軸

| 品質軸 | 評価基準 | 投資リサーチでの重要度 |
|--------|---------|-------------------|
| Point-in-time 対応 | 各時点でのセクター分類・指数構成が保持されているか | critical。現在の分類で過去を評価すると bias が生じる |
| 変更履歴の完全性 | セクター変更、指数構成変更、ティッカー変更の履歴が完全か | critical |
| ID マッピングの正確性 | 複数の ID 体系（ティッカー、ISIN、SEDOL、CUSIP）間の対応が正しいか | high。ID のミスマッチはデータ結合エラーの主因 |
| コーポレートアクションの完全性 | 分割・併合・配当の記録が完全か | critical。価格調整の基礎 |
| 上場廃止銘柄のカバレッジ | 廃止済み銘柄の過去データが保持されているか | critical。survivorship bias の根源 |

### 6. バイアス / 汚染 / 時間ずれリスク

| リスク ID | リスク名 | 説明 | 深刻度 |
|----------|---------|------|--------|
| MTA-B01 | セクター分類の遡及変更 | 現在のセクター分類を過去に適用（例: 2018年のGICS通信サービス再編を2010年に適用） | critical |
| MTA-B02 | 指数構成の survivorship | 現在の指数構成銘柄で過去のバックテストを実施 | critical |
| MTA-B03 | コーポレートアクションの欠落 | 株式分割や併合が記録されておらず、価格データが歪む | critical |
| MTA-B04 | ID マッピングの断絶 | ティッカー変更やM&A時のID断絶でデータが不連続に | high |
| MTA-B05 | 浮動株比率の遅延更新 | 実際の浮動株変動と反映タイミングのずれ | medium |

### 7. Leakage risk の典型パターン

| Leakage Pattern | Audit Rubric 接続 | 検出方法 |
|----------------|-------------------|---------|
| 現在の指数構成で過去のユニバースを構築 | LKG-02 (survivorship) | ユニバース構築方法の確認。point-in-time 指数構成を使用しているか |
| 現在のセクター分類で過去のファクター計算 | LKG-07 (point-in-time) | GICS 変更履歴と使用しているセクター分類の照合 |
| コーポレートアクション未反映の価格データ使用 | LKG-01 (look-ahead) | 調整係数の有無と正確性の確認 |
| 廃止銘柄の除外 | LKG-02 (survivorship) | ユニバースに廃止銘柄が含まれるか確認 |

### 8. Proxy として許容できる条件
- GICS 分類が必要だが東証33業種分類で代替 → **条件付き許容**。日本市場のみの分析で、セクターの粒度が十分な場合
- Point-in-time 指数構成が必要だが「現在の構成 + 廃止銘柄の手動追加」で代替 → **条件付き許容**。廃止銘柄の網羅性が検証できる場合のみ。quality_loss_estimate: medium

### 9. Proxy として許容できない条件
- Point-in-time 指数構成が必要だが現在の構成で代替 → **不可**。survivorship bias が直接混入
- コーポレートアクション履歴が必要だが「なし」で代替（分割等がないと仮定）→ **不可**。価格データの整合性が崩壊
- 複数 ID 体系のマッピングが必要だが「ティッカーのみ」で代替 → **不可**。ティッカーは再利用される。過去の異なる銘柄と混同するリスク

### 10. v1 で最低限検査すべき項目
- [ ] ユニバース構築に使用する指数構成の point-in-time 対応
- [ ] 上場廃止銘柄のカバレッジ
- [ ] コーポレートアクション履歴の完全性
- [ ] セクター分類の point-in-time 対応（分類変更の反映）
- [ ] ID マッピングの正確性（特に M&A / ティッカー変更時）

### 11. Audit Rubric との接続

| Audit Pattern | Metadata での検出対象 |
|--------------|---------------------|
| LKG-02 (survivorship) | ユニバース構築での廃止銘柄の扱い。指数構成の point-in-time 対応 |
| LKG-07 (point-in-time) | セクター分類の遡及適用。現在の分類で過去を評価していないか |
| EVD-01 | コーポレートアクションデータの入手可能性 |
| ASM-04 | Metadata の品質仮定（「ID マッピングは正しい」「分割は全て反映済み」等） |
| PRC-B01〜B02 | Price Data の調整精度は Metadata の正確性に依存。Metadata が不正確なら Price Data も信頼できない |

### 12. Validation Plan との接続

| Validation Test | Metadata の役割 |
|----------------|----------------|
| offline_backtest | ユニバース構築の基盤。point-in-time 対応が全テストの前提条件 |
| walk_forward | ユニバースの再構築タイミング（リバランス日の指数構成）の設定 |
| out_of_sample | サンプル外期間でのユニバース構成の確認 |
| regime_split | 各レジーム期間でのユニバース構成が正しいか |

## カテゴリ間依存関係

カテゴリは独立ではない。以下の依存関係がある。

```
[MTA] Metadata
  ↓ (ユニバース定義、ID マッピング、コーポレートアクション)
  ├── [PRC] Price Data       ← 調整済み価格は MTA に依存
  ├── [FND] Fundamental Data ← 銘柄特定、セクター分類は MTA に依存
  ├── [SNT] Sentiment Data   ← アナリストカバレッジの対象銘柄は MTA に依存
  └── [FLW] Flow Data        ← 銘柄・ファンドの特定は MTA に依存

[MAC] Macro Data
  ↓ (レジーム判定)
  ├── [PRC] Price Data       ← レジーム別分析の区間定義
  └── [FND] Fundamental Data ← 景気サイクルと業績サイクルの対応

[ALT] Alternative Data
  ↓ (独立性が高いが)
  └── [MTA] Metadata         ← 銘柄・企業へのマッピングに MTA が必要
```

**含意**: Metadata の品質問題は全カテゴリに伝播する。Audit において MTA の issue は他カテゴリの issue の根本原因である可能性がある。MTA の critical issue が存在する場合、PRC / FND / SNT / FLW の Audit 結果も信頼性が低下する。

## Appendix: Evidence Plan Object へのマッピング

v1_core_loop_spec.md の Evidence Plan Object の evidence_item に本 Taxonomy がどう接続するか:

```json
{
  "item_id": "string",
  "category": "price | fundamental | alternative | macro | sentiment | flow | metadata",
  "description": "string",
  "requirement_level": "required | optional | proxy_acceptable",
  "availability": "available | obtainable_with_effort | unavailable",
  "quality_concerns": ["string"],     // 本 Taxonomy の品質評価軸から
  "known_biases": ["string"],         // 本 Taxonomy のバイアスリスクIDを参照（例: "PRC-B01"）
  "temporal_coverage": {
    "start": "ISO-8601",
    "end": "ISO-8601",
    "frequency": "tick | minute | daily | weekly | monthly | quarterly | annual"
  },
  "proxy_option": {
    "description": "string",
    "quality_loss_estimate": "minimal | medium | severe",
    "permitted": true,               // 本 Taxonomy の proxy 許容条件に基づく
    "prohibition_reason": null       // 不可の場合、本 Taxonomy の proxy 不可条件を参照
  },
  "point_in_time_status": "full | partial | none",        // 新規追加フィールド
  "reporting_lag_days": null | number,                     // 新規追加フィールド
  "leakage_risk_patterns": ["string"]                      // Audit Rubric の LKG パターンID
}
```

**新規追加フィールド**: `point_in_time_status` と `reporting_lag_days` は本 Taxonomy の設計過程で必要性が判明した。internal_schema.md の Evidence Plan Object に追加を推奨する。

- `point_in_time_status`:
  - `full`: 各時点で利用可能だったデータバージョンが完全に保持されている
  - `partial`: 公表日情報はあるが、修正履歴は不完全
  - `none`: 最新値のみ。point-in-time 情報なし
- `reporting_lag_days`: データの基準日から公表日までの遅延日数。null = 遅延なし（リアルタイム）
